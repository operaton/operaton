## 1. Introduction & Purpose

This document is the single source of truth for writing API documentation in the Operaton REST API. It exists because an inconsistent API reference forces developers to guess, experiment, or read source code instead of shipping software. Every rule here exists to prevent that outcome.

### What This Guide Covers

- The FreeMarker template system used to generate the OpenAPI 3.0 specification
- All prose text that appears in operation summaries, descriptions, parameter descriptions, and response descriptions
- Conventions for tags, examples, deprecation notices, and cross-references
- A complete scaffold for adding a new endpoint

### What This Guide Does Not Cover

- REST API design decisions (URL structure, HTTP method choice, status code selection for new endpoints). Those require an [Architectural Decision Record](../decisions/).
- The generated Java client method names — these derive from `operationId` and **must not be changed** in documentation-only PRs.
- Response schema shapes — `$ref` targets are defined in DTO files outside the template system.

### The Golden Rule

**Every word in the documentation must serve the developer.** If a sentence does not tell the reader what to do, what to expect, or what went wrong, remove it.

## 2. API Documentation Philosophy

### Developer-First Mindset

The reader of this documentation is a backend engineer integrating with Operaton. They are:

- Working under time pressure
- Scanning, not reading
- Cross-referencing multiple tabs
- Trusting what they read (so be precise)

Write as if the developer is looking at the documentation for the first time, with a partially written HTTP client in front of them. Every description should let them complete that client without opening another document.

Practical consequences:
- Lead with the action, not the background
- State the happy path first, edge cases second
- When a parameter is required, say so in the parameter description too, not only in the `required` flag
- When a field only matters in a specific scenario, say when

### The Operaton BPMN Domain Model

Contributors must understand these terms to write accurate documentation. Use them exactly as defined — no synonyms, no abbreviations in prose.

| Term | Definition |
|---|---|
| **Process Definition** | The blueprint of a business process, parsed from a BPMN 2.0 XML file and stored after deployment. Identified by `id` (key:version:uuid) or `key`. Multiple versions can coexist. |
| **Process Instance** | A single running execution of a Process Definition. Starts when a process is triggered, ends when all tokens reach an end event or the instance is terminated. |
| **Task** | A human work item generated by a User Task element in a BPMN model. Assigned to a user or candidate group. Resides in the task inbox until claimed and completed. |
| **Execution** | The pointer that tracks the current position of a token inside a Process Instance. A process instance may have many concurrent executions (parallel branches). |
| **External Task** | A work item for an external worker process. The engine publishes the task; the worker fetches-and-locks it, performs work out of process, then reports success or failure. |
| **Decision Definition** | A DMN decision table or decision requirements graph deployed to the engine. Evaluated via the Decision Definition API or invoked from a Business Rule Task in a BPMN process. |
| **Incident** | An automatically created problem record when a job or external task exhausts its retries. Signals that operator intervention is required. |
| **Job** | An internal unit of deferred or scheduled work managed by the job executor (e.g., timer events, asynchronous continuations). |
| **Deployment** | The act of uploading one or more BPMN, CMMN, or DMN resources to the engine. Creates or updates Process Definitions, Decision Definitions, etc. |
| **Batch** | An asynchronous bulk operation (e.g., delete many process instances, set retries for many jobs). A batch runs in the background via jobs and reports completion count and error count. |

## 3. Summary Writing Rules

### The Formula

```
<Verb> <ResourceNoun>
```

The summary is the one-line label visible in API explorers, generated client documentation, and OpenAPI UIs. It must be immediately scannable.

### Length

**3–10 words.** No punctuation at the end. No articles ("the", "a", "an").

### Approved Verb Vocabulary

Use only the verbs in this table for the corresponding action. Never improvise synonyms.

| Verb | HTTP Methods | Usage Note |
|---|---|---|
| **List** | GET | Querying a collection with filters. Always use for the "get all" endpoint of a resource. |
| **Get** | GET | Fetching a single resource by identifier. |
| **Count** | GET | Returning only the numeric count matching a query. |
| **Create** | POST | Persisting a new resource. Use for `/create` sub-paths. |
| **Update** | PUT, PATCH | Replacing or partially updating a resource's data fields. |
| **Delete** | DELETE | Permanently removing a resource. |
| **Complete** | POST | Finishing a Task or External Task and advancing the process. |
| **Submit** | POST | Submitting a form — for Submit Task Form style operations. |
| **Claim** | POST | Assigning a task exclusively to the requesting user. |
| **Unclaim** | POST | Removing a user's exclusive claim from a task. |
| **Resolve** | POST | Marking an Incident or Task as resolved after operator intervention. |
| **Suspend** | PUT | Halting processing on a definition or instance without deleting it. |
| **Activate** | PUT | Re-enabling a previously suspended definition or instance. |
| **Start** | POST | Starting execution — for process instance start operations. |
| **Fetch** | POST | Retrieving items for worker processing (always paired with Lock). |
| **Lock** | POST | Reserving an item for exclusive processing. |
| **Unlock** | POST | Releasing an exclusive lock. |
| **Trigger** | POST | Sending a signal or message to advance a waiting execution. |
| **Validate** | POST | Checking validity without persisting anything. |
| **Set** | PUT | Assigning a specific value to a field that can be changed in isolation. |
| **Clear** | DELETE | Removing a specific value without deleting the parent resource. |
| **Deploy** | POST | Uploading resources to create deployments. (Prefer **Create** for `/deployment/create`.) |
| **Correlate** | POST | Matching an incoming message to one or more waiting process instances. |

### Summary Examples

Correct summaries for common Operaton endpoint patterns:

| Endpoint | Summary |
|---|---|
| `GET /process-definition` | `List Process Definitions` |
| `GET /process-definition/{id}` | `Get Process Definition` |
| `GET /process-definition/count` | `Count Process Definitions` |
| `POST /task/{id}/complete` | `Complete Task` |
| `POST /task/{id}/claim` | `Claim Task` |
| `POST /task/{id}/unclaim` | `Unclaim Task` |
| `POST /external-task/fetchAndLock` | `Fetch and Lock External Tasks` |
| `POST /deployment/create` | `Create Deployment` |
| `DELETE /process-definition/{id}` | `Delete Process Definition` |
| `PUT /process-definition/{id}/suspended` | `Suspend or Activate Process Definition` |
| `POST /process-instance/{id}/variables/{varName}` | `Set Process Instance Variable` |
| `GET /task/{id}` | `Get Task` |
| `POST /signal` | `Trigger Signal Event` |
| `GET /history/process-instance` | `List Historic Process Instances` |
| `POST /message` | `Correlate Message` |
| `GET /deployment/{id}` | `Get Deployment` |

### Anti-Patterns

- ❌ **Never include the HTTP method in the summary.** `(POST)`, `(GET)` — remove them entirely.
- ❌ **Never append "(Historic)"** to a summary. Historic endpoints already live under the `Historic *` tag.
- ❌ **Never use third-person verbs.** "Returns", "Creates", "Deletes" → write imperatively: "List", "Create", "Delete".
- ❌ **Never use articles** ("a", "an", "the") in the summary.
- ❌ **Never exceed 10 words.**

## 4. Description Writing Rules

### Voice and Person

Write in **active voice**, **second person** ("you"). The subject is always "you" (the developer calling the API), even when omitted.

- ✅ `Queries for process definitions that fulfill given parameters.`
- ✅ `Completes a task and updates process variables.`
- ❌ `This endpoint can be used to query process definitions.`
- ❌ `The caller provides a task ID and variables are updated.`

### Structure

Not every description needs all five layers. Short descriptions (1–2 sentences) are fine for simple endpoints. Add layers only when the information genuinely helps.

```
1. TL;DR sentence — what the endpoint does in one sentence.
2. When to use — optional; only include if there is a non-obvious use case
                 or when distinguishing from a similar endpoint.
3. Key behavior — optional; list notable side effects, ordering guarantees,
                  atomicity notes, or filtering behavior.
4. Security note — required whenever the endpoint can execute custom code,
                   modify access controls, or expose sensitive data.
5. Cross-references — link to count endpoint, POST query variant, or
                      conceptual guide using ${docsUrl}.
```

### Prohibited Patterns

- ❌ **No Java class or interface names.** Never write `ProcessDefinition interface`, `RuntimeService`, `HistoryService`, `TaskServiceImpl`, etc. Developers consuming the REST API do not use the Java API.
- ❌ **No "This endpoint..."** — omit the subject. Start with the verb.
- ❌ **No restating the HTTP method.** ("This POST endpoint...", "Making a GET request...")
- ❌ **No marketing language.** ("powerful", "flexible", "easy to use")
- ❌ **No stating the obvious.** ("Returns HTTP 200 on success." — put that in the response description.)

### Description Examples

---

#### `GET /process-definition` — List Process Definitions

```
Queries for process definitions that fulfill given parameters. Filter by name, key, version,
deployment, tenant, or version tag. Results are paginated using `firstResult` and `maxResults`.

Use the [Get List Count](${docsUrl}/reference/rest/process-definition/get-query-count/) endpoint
to retrieve the total number of matching definitions without fetching the full list.
```

---

#### `GET /process-definition/{id}` — Get Process Definition

```
Retrieves a single process definition by its unique `id`.

The `id` has the format `key:version:uuid` (e.g., `invoice:1:c3a63aaa-2046-11e7-8f94-34f39ab71d4e`).
To look up a definition by key instead, use the
[Get by Key](${docsUrl}/reference/rest/process-definition/get/) variant.
```

---

#### `POST /task/{id}/complete` — Complete Task

```
Completes a task and advances the process to the next step.

Submit process variables in the request body to update them as part of completion. If
`withVariablesInReturn` is set to `true`, the response includes all process variables visible
in the scope at the time of completion. Otherwise, the response body is empty (HTTP 204).

The task must be in an uncompleted state and must belong to an active process instance.
```

---

#### `POST /external-task/fetchAndLock` — Fetch and Lock External Tasks

```
Fetches and locks a batch of external tasks for a worker to execute.

Specify the worker identifier, the maximum number of tasks to fetch, and one or more topic
subscriptions. Each topic subscription can define its own lock duration (in milliseconds),
variable filter, tenant filter, and process definition filter.

Locked tasks are exclusively reserved for the worker until the lock expires. Workers must
call the complete, report-failure, or extend-lock endpoints before expiry.

To control fetch order, set `usePriority` to `true` (higher priority tasks fetched first) or
provide a `sorting` array to order by `createTime` or other supported fields.

See the [External Task guide](${docsUrl}/user-guide/process-engine/external-tasks/) for the
complete worker pattern.
```

---

#### `POST /deployment/create` — Create Deployment

```
Deploys one or more BPMN, CMMN, or DMN resources to the engine.

Upload resources as `multipart/form-data` fields. Include `deployment-name`,
`deployment-source`, and optionally `tenant-id` and `enable-duplicate-filtering` fields
alongside the resource files.

When duplicate filtering is enabled, the engine skips deployment if the resources are
identical to those in the most recent deployment with the same name and source.

**Security consideration:** Deployments can contain scripts and EL expressions that execute
on the server. Restrict access to this endpoint to trusted operators only. Do not expose it
to end users.
```

---

### Description Anti-Patterns Summary

| Anti-Pattern | Example | Fix |
|---|---|---|
| Java class name | `according to the ProcessDefinition interface` | Remove class reference entirely |
| This endpoint | `This endpoint allows you to...` | Start with the verb directly |
| Passive voice | `A task is completed and variables are updated` | `Completes a task and updates variables` |
| Restating HTTP method | `Making a GET request to this URL...` | Omit; start with the action |
| Marketing filler | `This powerful endpoint provides flexible querying` | Remove; state what it does |
| "In order to" | `In order to fetch tasks, provide...` | `To fetch tasks, provide...` |
| "Please note" | `Please note that the task must be active` | `The task must be active` |

## 5. Parameter Documentation Rules

### Format Convention

Every parameter description is written as a **complete sentence** ending with a period. Start with what the parameter is, not "This parameter is".

```freemarker
<@lib.parameter
    name = "id"
    location = "path"
    type = "string"
    required = true
    last = true
    desc = "The id of the process definition to retrieve." />
```

For query parameters with significant filtering semantics, use a fuller description:

```freemarker
desc = "Filter by the version tag. An exact match is performed on the version tag string." 
```

### Required vs Optional

- State required parameters explicitly in the description: `"Required. The id of the task to complete."`
  *(The `required = true` flag is necessary but insufficient — some OpenAPI renderers do not highlight it visually.)*
- Optional parameters do not need the word "optional" unless the omission behaviour is surprising. In that case: `"Optional. When omitted, all tenants are included."`

### ID Parameters

Always explain:
1. What format the ID takes
2. How the caller obtains the ID

```freemarker
desc = "The id of the process definition. The format is \`key:version:uuid\`,
        for example \`invoice:1:c3a63aaa-2046-11e7-8f94-34f39ab71d4e\`.
        Retrieve the id from the [List Process Definitions](${docsUrl}/reference/rest/process-definition/get-query/)
        endpoint."
```

For process instance IDs:

```freemarker
desc = "The id of the process instance. Retrieve it from the response of
        [Start Process Instance](${docsUrl}/reference/rest/process-definition/post-start-process-instance/)
        or from the [List Process Instances](${docsUrl}/reference/rest/process-instance/get-query/) endpoint."
```

### Date Parameters

Always state the format explicitly. Always include the timezone note.

```freemarker
desc = "Restrict to tasks created after the given date. The date must be in
        [ISO 8601](https://en.wikipedia.org/wiki/ISO_8601) format, including
        timezone offset. Example: \`2025-01-15T10:00:00.000+0100\`."
```

### Enum Parameters

List all accepted values with their effect. Do not rely solely on the `enumValues` array — that produces only a validation rule, not understanding.

```freemarker
<@lib.parameter
    name = "sortOrder"
    location = "query"
    type = "string"
    enumValues = ['"asc"', '"desc"']
    desc = "Sort results in ascending (\`asc\`) or descending (\`desc\`) order.
            Must be used together with \`sortBy\`." />
```

For `sortBy` parameters, list what each value sorts by in a way the developer can understand without reading source code:

```freemarker
desc = "Sort results by a given criterion. Valid values: \`id\` (task id),
        \`name\` (task name), \`assignee\` (assigned user id), \`created\` (creation date),
        \`due\` (due date), \`priority\` (numeric priority). Must be used together with
        \`sortOrder\`."
```

### Boolean Parameters

State both the `true` and `false` (or default) effects explicitly. Never write just "Whether to include X."

```freemarker
<@lib.parameter
    name = "cascade"
    location = "query"
    type = "boolean"
    desc = "If \`true\`, all running process instances, historic process instances,
            and jobs for this process definition are deleted along with the definition.
            If \`false\` (default), only the process definition record is deleted;
            existing instances are preserved." />
```

### Filter-by-List Parameters

When a parameter accepts a comma-separated list, state this:

```freemarker
desc = "Filter by a comma-separated list of tenant ids. A process definition is included
        if it belongs to any of the listed tenants. Example: \`tenantA,tenantB\`."
```

## 6. Response Documentation Rules

### What to Write per Status Code

#### `200 OK`

Write one short sentence describing what the body contains. If the body is an array, say so.

- ✅ `Request successful. Returns an array of process definition objects.`
- ✅ `Request successful. Returns the process definition with the given id.`
- ❌ `The request was processed successfully and the server returned the expected data.`

Always include `examples` for 200 responses. See [Section 8](#8-examples-policy).

#### `204 No Content`

State that no body is returned. State why (the operation was a side effect with no resource to return).

- ✅ `Request successful. No content is returned.`
- ✅ `Request successful. The task was completed; no variables are returned because \`withVariablesInReturn\` was not set.`

#### `400 Bad Request`

Describe the specific invalid condition. Always link to the error response overview.

- ✅ `Returned if some of the query parameters are invalid, for example if a \`sortOrder\` parameter is supplied but no \`sortBy\`. See the [Introduction](${docsUrl}/reference/rest/overview/#error-handling) for the error response format.`
- ✅ `The variable value or type is invalid, for example if the value could not be parsed to an Integer or the variable type is not supported.`
- ❌ `Bad request.`
- ❌ `The request is invalid.`

#### `401 Unauthorized`

**Copy-paste boilerplate:**

```freemarker
<@lib.response
    code = "401"
    dto = "ExceptionDto"
    desc = "Not authenticated. See the
            [Introduction](${docsUrl}/reference/rest/overview/#authentication)
            for the error response format." />
```

#### `403 Forbidden`

**Copy-paste boilerplate:**

```freemarker
<@lib.response
    code = "403"
    dto = "ExceptionDto"
    desc = "Not authorized to perform this operation. See the
            [Introduction](${docsUrl}/reference/rest/overview/#authorization)
            for the error response format." />
```

#### `404 Not Found`

Always name the missing resource and the identifier used to look it up:

- ✅ `Process definition with the given \`id\` does not exist. See the [Introduction](${docsUrl}/reference/rest/overview/#error-handling) for the error response format.`
- ✅ `No task with the given \`id\` exists.`
- ❌ `Resource not found.`
- ❌ `The process definition does not exist.` *("with the given id" must appear)*

#### `409 Conflict`

Describe the conflict condition:

- ✅ `Returned if the task is already claimed by a different user. See the [Introduction](${docsUrl}/reference/rest/overview/#error-handling) for the error response format.`

#### `500 Internal Server Error`

Describe the engine-level failure scenario:

- ✅ `If the task does not exist or the corresponding process instance could not be resumed successfully. See the [Introduction](${docsUrl}/reference/rest/overview/#error-handling) for the error response format.`
- ❌ `An internal server error occurred.`

### The `last = true` Rule

The last `@lib.response` call in each endpoint template **must** include `last = true` to suppress the trailing comma in generated JSON. Every other response call must omit it. Forgetting this causes a JSON parse error in the generated spec.

## 7. HTTP Status Code Semantics Reference

| Code | Meaning | When to Use | Operaton-Specific Notes |
|---|---|---|---|
| **200** | OK | Successful read or action returning a body. | Use for: list, get, count, complete (when `withVariablesInReturn=true`), fetchAndLock, start process instance. |
| **204** | No Content | Successful action with no response body. | Use for: delete, suspend/activate, unclaim, set variable, complete (default, no variables in return), unlock. |
| **400** | Bad Request | The request is structurally invalid, has missing required fields, has invalid parameter combinations, or a BPMN resource cannot be parsed. | Invalid `sortBy`/`sortOrder` combination; invalid variable type; BPMN parse error (use `ParseExceptionDto`). **Not** for "resource not found." |
| **401** | Unauthorized | The caller is not authenticated. | Always use the standard authentication link to the Introduction. |
| **403** | Forbidden | The caller is authenticated but does not have the required permission. | Use when Operaton's authorization system (`Authorization` resource) denies the operation. |
| **404** | Not Found | The specific resource identified by the request does not exist. | Name the resource type and the identifier in the description. **Not** for malformed requests — that is 400. |
| **409** | Conflict | The operation cannot be completed because of a resource state conflict. | Task already claimed by another user; trying to start an instance of a suspended process definition. |
| **500** | Internal Server Error | An unexpected engine-level failure occurred. | Use when the engine throws an exception that is not a client error (e.g., process execution failure, job executor error, task not found when completing). |

## 8. Examples Policy

### When Examples Are Required

| Endpoint Type | Request Body Example | 200 Response Example |
|---|---|---|
| POST / PUT / PATCH with request body | **Required** | Required if 200 is returned |
| GET list endpoint | Not applicable | **Required** |
| GET single resource | Not applicable | **Required** |
| DELETE | Not applicable | Not applicable (204) |
| POST with no body | Not applicable | Required if 200 is returned |

### Example Summary Formula

The `summary` field in an example must identify the request:

```
<HTTP METHOD> `<path>`[( <disambiguating label> )]
```

Examples:
- `"summary": "GET /process-definition?keyLike=invoice&sortBy=version&sortOrder=asc"`
- `"summary": "POST /external-task/fetchAndLock (with variable filtering)"`
- `"summary": "POST /external-task/fetchAndLock (with extension properties)"`
- `"summary": "Status 200 response"` — acceptable for simple GET single-resource examples

When a single endpoint has multiple examples (like fetchAndLock), append a disambiguating label in parentheses. Number them when the label would be too long: `(1)`, `(2)`, `(3)`.

### Realistic IDs

Do not use made-up strings like `"myId"` or `"someProcessDefinitionId"` for ID fields. Use IDs that match the real format:

| Field | Correct Example Format |
|---|---|
| Process Definition `id` | `"invoice:1:c3a63aaa-2046-11e7-8f94-34f39ab71d4e"` |
| Process Instance `id` | `"3b2a1c9d-4e5f-11ee-be56-0242ac120002"` |
| Task `id` | `"7f3a4b2e-1c9d-4e5f-a1b2-34f39ab71d4e"` |
| Deployment `id` | `"c398cd26-2046-11e7-8f94-34f39ab71d4e"` |
| External Task `id` | `"anExternalTaskId"` *(opaque string — generic placeholder acceptable)* |
| Worker `id` | `"aWorkerId"` *(application-defined — generic placeholder acceptable)* |
| Lock duration | `10000` *(milliseconds — always use a realistic positive integer)* |
| Timestamp | `"2015-10-06T16:34:42.000+0200"` *(ISO 8601 with timezone offset)* |

### Variable Type Format

Variables in examples must use the Operaton typed-value format:

```json
"variables": {
  "orderId": {
    "type": "String",
    "value": "INV-2025-001",
    "valueInfo": {}
  },
  "approved": {
    "type": "Boolean",
    "value": true,
    "valueInfo": {}
  },
  "amount": {
    "type": "Double",
    "value": 42.50,
    "valueInfo": {}
  }
}
```

The short-form (value only, no type wrapper) is acceptable only for `CompleteTaskDto`-style bodies where variables are submitted without type metadata:

```json
"variables": {
  "aVariable": { "value": "aStringValue" },
  "anotherVariable": { "value": 42 }
}
```

### Model: fetchAndLock

The `POST /external-task/fetchAndLock` endpoint is the gold standard for multi-example documentation. Study `paths/external-task/fetchAndLock/post.ftl` before writing any complex POST endpoint. Note:
- Four request examples, each demonstrating a distinct configuration
- Disambiguating labels in parentheses: `(1)`, `(2)`, `(3)`, `(4)`
- Response examples use the full DTO shape, not minimal stubs
- `null` is used explicitly where a field is present but empty (not omitted)

## 9. Deprecation Rules

### Deprecating an Entire Operation

Set `deprecated = true` in `@lib.endpointInfo`. The description must include a deprecation notice explaining why and what to use instead.

```freemarker
<@lib.endpointInfo
    id = "getTelemetryData"
    tag = "Telemetry"
    summary = "Fetch Telemetry Data"
    deprecated = true
    desc = "**Deprecated:** Telemetry collection has been removed as of Operaton 1.1.
            This endpoint will always return an empty response. It will be removed in a future release." />
```

### Deprecating a Field in a DTO

Set `deprecated = true` in `@lib.property`:

```freemarker
<@lib.property
    name = "caseInstanceId"
    type = "string"
    deprecated = true
    desc = "**Deprecated:** CMMN is not supported. This field is always \`null\`." />
```

### Deprecation Description Text Formula

```
**Deprecated:** <reason for deprecation>. Use <alternative endpoint or field> instead.
```

Or, when there is no replacement:

```
**Deprecated:** <reason for deprecation>. This field is always \`null\` and will be removed in a future release.
```

### CMMN Fields Rule

All fields named `caseInstanceId`, `caseExecutionId`, `caseDefinitionId`, `caseDefinitionKey`, or any other field specifically for CMMN (Case Management Model and Notation) must be marked `deprecated = true` with the description:

```
**Deprecated:** CMMN support has been removed. This field is always \`null\`.
```

This applies across all DTOs and response schemas where these fields appear.

## 10. Cross-Reference Patterns

All cross-references use `${docsUrl}` to resolve to the versioned documentation base:

```
${docsUrl} = https://docs.operaton.org/manual/${docsVersion}
```

### Pattern 1: List → Count Endpoint

Include at the end of every **List** endpoint description:

```
Use the [Get List Count](${docsUrl}/reference/rest/<resource>/get-query-count/) endpoint
to retrieve the total number of matching <resources> without fetching the full list.
```

**Examples:**
- Process Definition List → `[Get List Count](${docsUrl}/reference/rest/process-definition/get-query-count/)`
- Task List → `[Get Task Count](${docsUrl}/reference/rest/task/get-query-count/)`
- External Task List → `[Get External Task Count](${docsUrl}/reference/rest/external-task/get-query-count/)`

### Pattern 2: GET Query → POST Query Variant

When a `GET` list endpoint has a `POST` counterpart that accepts a request body for complex filters:

```
For more complex queries that exceed URL length limits, use the
[List (POST)](${docsUrl}/reference/rest/<resource>/post-query/) variant that accepts
filter criteria in the request body.
```

### Pattern 3: Single Resource → Key Variant

When both an `/{id}` and a `/key/{key}` variant exist:

In the `/{id}` endpoint:
```
To look up by process definition key instead of id, use the
[Get by Key](${docsUrl}/reference/rest/process-definition/get-by-key/) endpoint.
```

## 11. Tag Description Rules

### Required Fields per Tag

Every tag entry in `main.ftl` must eventually have:

```json
{
  "name": "Process Definition",
  "description": "<full tag description>",
  "externalDocs": {
    "description": "Find out more",
    "url": "${docsUrl}/reference/rest/process-definition/"
  }
}
```

*(Currently, `main.ftl` emits only `{"name": "..."}` entries. Adding `description` and `externalDocs` is the next enrichment target.)*

### Formula for Tag Description Content

1. **What this resource is** (1 sentence, domain definition)
2. **What operations are available** (1 sentence listing the primary actions)
3. **Where it fits in the BPMN lifecycle** (optional, for core concepts)

Length: 2–4 sentences. No bullet lists in tag descriptions.

---

### Complete Tag Descriptions

#### Authorization
Manages access control entries (ACEs) that grant or revoke permissions for users and groups on specific resources. Use these endpoints to create, update, delete, and query authorization rules. For an overview of the permission model, see the Authorization section of the user guide.

#### Batch
Represents an asynchronous bulk operation running in the background via the job executor, such as bulk deletion of process instances or bulk retry of jobs. Endpoints provide visibility into batch progress, pause or resume execution, and retrieve historic batch records.

#### Condition
Provides an endpoint to trigger evaluation of condition-based start events. Send a condition variable map; the engine evaluates all condition start events and starts matching process instances.

#### Decision Definition
Represents a deployed DMN decision table or decision literal expression. Endpoints support listing, retrieving, evaluating, and querying the XML source of decision definitions. Decision definitions are created by deploying DMN resources.

#### Decision Requirements Definition
Represents a DMN Decision Requirements Graph (DRG), a collection of related decision definitions with explicit dependency relationships. Endpoints support listing, retrieving, and retrieving the XML and diagram of decision requirements definitions.

#### Deployment
Represents a single deployment operation that uploaded one or more BPMN, CMMN, or DMN resources to the engine. Endpoints support creating new deployments, listing and retrieving existing deployments, and deleting deployments. Creating a deployment parses resources and registers the contained definitions.

#### Engine
Provides a single endpoint that returns the names of all process engines available in the current installation. Use this to discover available named engines for multi-engine setups.

#### Event Subscription
Represents a registered subscription to a BPMN event (message, signal, conditional, or compensate). Endpoints support listing and counting active event subscriptions across all process instances.

#### Execution
Represents a pointer tracking the current position of a token within a process instance. A single process instance can have multiple concurrent executions when parallel flows are active. Endpoints support querying executions, retrieving and setting local variables on executions, and triggering message event subscriptions.

#### External Task
Represents a service task configured for external handling: the engine publishes the task, and an external worker process fetches, locks, and completes it. Endpoints support the full external task lifecycle: fetch-and-lock, complete, report failure, report BPMN error, extend lock, and unlock.

#### Filter
Represents a saved task query that can be executed repeatedly, optionally shared with groups. Endpoints support creating, updating, deleting, and executing filters. Used primarily by the Operaton Tasklist web application.

#### Group
Represents a user group in Operaton's built-in identity service. Endpoints support creating, retrieving, updating, and deleting groups, and managing group membership. For LDAP-backed identity, most write operations are unavailable.

#### Historic Activity Instance
Provides access to the audit history of individual activity instances — that is, every time a BPMN activity was entered and exited during process execution. Supports list, count, and single-record retrieval with rich filtering options.

#### Historic Batch
Provides access to the audit history of completed or ongoing batch operations. Supports listing and counting historic batch records, and cleaning up finished batches.

#### Historic Decision Definition
Provides statistics about historic decision evaluations grouped by decision definition. Supports retrieving the count of historic decision instances per definition.

#### Historic Decision Instance
Provides access to the complete audit history of every DMN decision evaluation: input values, output values, rules matched, and evaluation time. Supports list, count, single-record retrieval, and bulk deletion.

#### Historic Decision Requirements Definition
Provides access to historic decision requirements definitions, including the count of decision instances evaluated under each requirements graph.

#### Historic Detail
Provides access to fine-grained audit events recorded during process execution: variable updates, form fields submitted, and other detail records. Supports list, count, and single-record retrieval with extensive filter options.

#### Historic External Task Log
Provides access to the audit log of external task lifecycle events: creation, failure, success, deletion, and lock expiration. Supports list, count, and single-record retrieval.

#### Historic Identity Link Log
Provides access to the history of identity link changes on tasks and process definitions: candidate user/group additions and removals, assignee changes. Supports list and count.

#### Historic Incident
Provides access to the audit history of incidents: when they were created, when they were resolved or deleted, and their cause. Supports list and count.

#### Historic Job Log
Provides access to the audit history of job execution events: creation, execution, failure, and deletion. Each entry captures the job's configuration and outcome at the time of the event. Supports list, count, and single-record retrieval including stack trace retrieval.

#### Historic Process Definition
Provides statistics about process definitions based on historic data, including the number of instances in each state (active, suspended, completed, externally terminated, internally terminated). Also provides access to cleanable process instance reports.

#### Historic Process Instance
Provides access to the complete audit history of all process instances, including those that have finished. Supports list, count, single-record retrieval, deletion, and bulk variable retrieval.

#### Historic Task Instance
Provides access to the audit history of task instances: creation, assignment, completion, and deletion events. Supports list, count, and single-record retrieval with extensive filter options.

#### Historic User Operation Log
Provides access to the audit log of operations performed by users through the API (deployments, task completions, variable modifications, suspension state changes, etc.). Supports list, count, and annotation of individual log entries.

#### Historic Variable Instance
Provides access to the last known value of every process variable for historic process instances. Supports list, count, and retrieval of binary variable data.

#### History Cleanup
Manages the history cleanup job that removes historic data from the database based on configured time-to-live (TTL) settings. Endpoints support triggering cleanup, querying cleanup configuration, and retrieving cleanup job details.

#### Identity
Provides utility endpoints for the identity subsystem: retrieving group information for the current user and validating passwords against the configured password policy.

#### Incident
Represents a problem record automatically created when a job or external task exhausts its retries, or created manually. Endpoints support listing, counting, retrieving, creating, deleting, and resolving incidents.

#### Job
Represents an internal unit of work managed by the job executor: timer events, asynchronous continuations, and message correlations. Endpoints support listing, retrieving, executing, retrying, suspending, and configuring jobs.

#### Job Definition
Represents the configuration of a class of jobs associated with a specific activity in a process definition. Endpoints support listing, retrieving, and modifying retry configuration, priority, and suspension state for all jobs of a given type.

#### Message
Provides a single endpoint for correlating a named BPMN message to one or more waiting process instances or start events. Used to trigger message start events and resume processes waiting at intermediate message catch events.

#### Metrics
Provides time-series metrics about engine activity: number of activity instances, job executions, and decision evaluations. Supports aggregated queries by time range and returns sum metrics per metric name.

#### Migration
Provides endpoints for migrating running process instances from one version of a process definition to another. Supports generating migration plans, validating plans, and executing migrations synchronously or asynchronously.

#### Modification
Provides endpoints for modifying the token flow of running process instances: starting before or after activities, and cancelling active activity instances. Supports synchronous and asynchronous (batch) execution.

#### Process Definition
Represents a deployed BPMN process definition — the blueprint for creating process instances. Endpoints support listing, retrieving, starting instances, retrieving XML source and diagram images, suspending and activating, deleting, and querying statistics.

#### Process Instance
Represents a single running execution of a process definition. Endpoints support listing, retrieving, deleting, modifying variables, retrieving activity instances, suspending and activating, and bulk operations via batch.

#### Schema Log
Provides an endpoint to retrieve the history of database schema updates applied by Liquibase. Useful for verifying the database schema version in deployed installations.

#### Signal
Provides a single endpoint for throwing a named BPMN signal event to all active signal event handlers. Signals have global broadcast semantics — all matching handlers receive the signal simultaneously.

#### Task
Represents a human task assigned to a user or candidate group. Endpoints support the full task lifecycle: listing, retrieving, creating, updating, delegating, claiming, unclaiming, completing, resolving, submitting task forms, and managing task-level variables, attachments, comments, and identity links.

#### Task Attachment
Manages file attachments on individual tasks. Endpoints support creating, retrieving, listing, and deleting attachments, as well as retrieving attachment binary content.

#### Task Comment
Manages text comments on individual tasks. Endpoints support creating, retrieving, listing, and deleting comments. Comments provide a human-readable audit trail of task activity.

#### Task Identity Link
Manages identity links on tasks: relationships between a task and a user or group (candidate, assignee, owner). Endpoints support adding, removing, and listing identity links.

#### Task Local Variable
Manages variables that are scoped exclusively to a single task (not visible to the process instance). Endpoints support listing, getting, setting, and deleting local variables, including binary and file variables.

#### Task Variable
Manages variables visible in the scope of a task, including variables from the enclosing process instance scope. Endpoints support listing, getting, setting, and deleting variables, including binary and file variables.

#### Telemetry
**Deprecated.** Telemetry data collection has been removed from Operaton. These endpoints are retained for backwards compatibility but return empty or no-op responses. They will be removed in a future release.

#### Tenant
Represents a tenant in a multi-tenancy deployment. Endpoints support creating, retrieving, updating, and deleting tenants, and managing tenant membership for users and groups.

#### User
Represents a user in Operaton's built-in identity service. Endpoints support creating, retrieving, updating, and deleting users, managing user credentials, and unlocking locked user accounts. For LDAP-backed identity, most write operations are unavailable.

#### Variable Instance
Represents a live process variable in a running process instance, execution, task, or case execution. Endpoints support listing, counting, and retrieving variable instances and their binary data.

#### Version
Provides a single endpoint that returns the version number of the Operaton engine currently deployed. Useful for compatibility checks and health verification.

## 13. FreeMarker Template Quick Reference

### The Five Core Macros

#### 1. `@lib.endpointInfo` — Operation Metadata

```freemarker
<@lib.endpointInfo
    id         = "operationId"      <#-- REQUIRED. Generates Java client method name. NEVER change. -->
    tag        = "Tag Name"         <#-- REQUIRED. Must exactly match a tag name in main.ftl. -->
    summary    = "Verb Noun"        <#-- REQUIRED. 3-10 words, imperative. See Section 3. -->
    desc       = "Description."     <#-- REQUIRED. Full prose. Supports Markdown. See Section 4. -->
    deprecated = false />           <#-- Optional. Default: false. Set true for deprecated ops. -->
```

#### 2. `@lib.parameter` — Query or Path Parameter

```freemarker
<@lib.parameter
    name         = "paramName"       <#-- REQUIRED. camelCase for query, kebab-case for path. -->
    location     = "query"           <#-- REQUIRED. "query" | "path" | "header" -->
    type         = "string"          <#-- REQUIRED. "string" | "boolean" | "integer" | "array" -->
    desc         = "Description."    <#-- REQUIRED. Complete sentence. See Section 5. -->
    required     = false             <#-- Optional. Default: false. -->
    enumValues   = []                <#-- Optional. List of quoted string literals. -->
    defaultValue = ""                <#-- Optional. Unquoted for boolean/integer, quoted for string. -->
    format       = ""                <#-- Optional. e.g. "date-time", "int64" -->
    last         = false />          <#-- REQUIRED on final parameter. Suppresses trailing comma. -->
```

#### 3. `@lib.requestBody` — POST/PUT Request Body

```freemarker
<@lib.requestBody
    mediaType   = "application/json"  <#-- REQUIRED. Usually "application/json" or "multipart/form-data" -->
    dto         = "MyRequestDto"      <#-- REQUIRED if mediaType is application/json -->
    requestDesc = ""                  <#-- Optional. Additional context about the request body. -->
    examples    = [] />               <#-- Optional but REQUIRED for all POST/PUT. See Section 8. -->
```

#### 4. `@lib.response` — HTTP Response

```freemarker
<@lib.response
    code                = "200"         <#-- REQUIRED. HTTP status code as string. -->
    desc                = "Description" <#-- REQUIRED. See Section 6. -->
    dto                 = "MyDto"       <#-- Required unless code is "204" or flatType is set. -->
    array               = false         <#-- Optional. true if response is an array of dto. -->
    additionalProperties = false        <#-- Optional. true for map-shaped responses. -->
    flatType            = ""            <#-- Optional. Use for "string" or other primitives. -->
    mediaType           = "application/json" <#-- Optional. Default: application/json -->
    binary              = false         <#-- Optional. true for file download responses. -->
    examples            = []            <#-- Required for 200 responses. See Section 8. -->
    last                = false />      <#-- REQUIRED on final response. Suppresses trailing comma. -->
```

#### 5. `@lib.property` — DTO Field

```freemarker
<@lib.property
    name               = "fieldName"   <#-- REQUIRED. camelCase. -->
    type               = "string"      <#-- REQUIRED. "string" | "boolean" | "integer" | "number" | "array" | "object" | "ref" -->
    desc               = "Description" <#-- REQUIRED. -->
    enumValues         = []            <#-- Optional. -->
    defaultValue       = ""            <#-- Optional. -->
    nullable           = true          <#-- Optional. Default: true -->
    minimum            = ""            <#-- Optional. For numeric types. -->
    deprecated         = false         <#-- Optional. Default: false. -->
    additionalProperties = false       <#-- Optional. For map-typed fields. -->
    itemType           = "string"      <#-- Optional. Element type for array fields. -->
    dto                = ""            <#-- Optional. $ref target for array or ref types. -->
    format             = ""            <#-- Optional. e.g. "date-time", "int64" -->
    last               = false />      <#-- REQUIRED on final property. -->
```

### Scaffold for a New Endpoint

Use this as the starting template for every new endpoint file at
`paths/<resource>/<sub-path>/<method>.ftl`:

```freemarker
<#macro endpoint_macro docsUrl="">
{

  <@lib.endpointInfo
      id      = "operationId"
      tag     = "Resource Tag"
      summary = "Verb Noun"
      desc    = "One-sentence TL;DR of what this endpoint does.

                 Additional detail paragraph if needed. Use Markdown.
                 Reference: [Related Concept](${docsUrl}/user-guide/process-engine/.../)." />

  "parameters" : [

    <@lib.parameter
        name     = "id"
        location = "path"
        type     = "string"
        required = true
        last     = true
        desc     = "The id of the resource." />

  ],

  <@lib.requestBody
      mediaType = "application/json"
      dto       = "MyRequestDto"
      examples  = ['"example-1": {
                     "summary": "POST /resource/{id}/action",
                     "description": "Description of this example.",
                     "value": {
                       "fieldName": "fieldValue"
                     }
                   }'] />

  "responses" : {

    <@lib.response
        code     = "204"
        desc     = "Request successful." />

    <@lib.response
        code     = "400"
        dto      = "ExceptionDto"
        desc     = "Returned if <specific invalid condition>. See the
                    [Introduction](${docsUrl}/reference/rest/overview/#error-handling)
                    for the error response format." />

    <@lib.response
        code     = "401"
        dto      = "ExceptionDto"
        desc     = "Not authenticated. See the
                    [Introduction](${docsUrl}/reference/rest/overview/#authentication)
                    for the error response format." />

    <@lib.response
        code     = "403"
        dto      = "ExceptionDto"
        desc     = "Not authorized to perform this operation. See the
                    [Introduction](${docsUrl}/reference/rest/overview/#authorization)
                    for the error response format." />

    <@lib.response
        code     = "404"
        dto      = "ExceptionDto"
        last     = true
        desc     = "Resource with the given \`id\` does not exist. See the
                    [Introduction](${docsUrl}/reference/rest/overview/#error-handling)
                    for the error response format." />

  }
}

</#macro>
```

**Steps to use this scaffold:**

1. Copy the scaffold to the correct path under `paths/`
2. Replace `operationId` — match the agreed Java client method name (verify with the team)
3. Replace `"Resource Tag"` — must exactly match a tag name in `main.ftl`
4. Write the summary (3–10 words, approved verb)
5. Write the description (active voice, second person, no Java class names)
6. Add parameters (path params first, then query params, then `last = true` on the final one)
7. Remove `@lib.requestBody` block if the endpoint has no request body
8. Replace 204 with 200 + DTO if the endpoint returns a response body
9. Remove status codes that do not apply
10. Ensure the final `@lib.response` has `last = true`
11. Register the path in the `endpoints` map in `main.ftl`

### Using `${docsUrl}` Correctly

`${docsUrl}` is injected into every template by the `main.ftl` rendering loop:

```freemarker
<#assign docsUrl = "https://docs.operaton.org/manual/${docsVersion}">
```

**Always** use `${docsUrl}` for documentation links. Never hardcode `https://docs.operaton.org/manual/7.22/` or any other hardcoded version.

Correct:
```freemarker
desc = "See the [Introduction](${docsUrl}/reference/rest/overview/#error-handling)."
```

Incorrect:
```freemarker
desc = "See the [Introduction](https://docs.operaton.org/manual/7.22/reference/rest/overview/#error-handling)."
```

### Common Template Mistakes

| Mistake | Symptom | Fix |
|---|---|---|
| Missing `last = true` on final `@lib.response` | Generated JSON has a trailing comma → spec parse error | Add `last = true` to the last response call |
| Missing `last = true` on final `@lib.parameter` | Same parse error | Add `last = true` to the last parameter call |
| Hardcoded docs URL | Links break on version upgrades | Use `${docsUrl}` |
| Backtick not escaped | FreeMarker renders literal `\`` | Use `\`` in desc strings |
| Newlines in desc not indented | `removeIndentation` strips indentation, leaving merged lines | Indent continuation lines with at least 2 spaces (they will be cleaned) |
| Wrong tag name | Endpoint orphaned in "Other" in UI | Tag must exactly match a name in the `main.ftl` tags array |
| Changing `operationId` | Java client method name changes, breaking existing clients | Never change `id` in `@lib.endpointInfo` |
| Using `"last": true` (JSON style) | FreeMarker will not process it | Use `last = true` (FreeMarker parameter style) |
